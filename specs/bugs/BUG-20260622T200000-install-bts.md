# BUG-20260622T200000: install.sh does not install bts itself

## Problem

Running `install.sh` sets up the entire toolchain (ripgrep, fd, bat, aider, etc.) but does not install the `bts` dispatcher itself. After a fresh pipe-mode install:

```
$ bts --version
bts: command not found
```

**Actual**: `bin/bts` lives in the repository but is never copied to a system-accessible location. Users must clone the repo manually to use `bts`.

**Expected**: `bts` is available on `PATH` immediately after `curl | bash` — no clone needed.

**Reproduce**: `curl -fsSL https://raw.githubusercontent.com/danielvm-git/big-token-saver/main/install.sh | bash && bts --version`

## Root Cause Analysis

### Reproduce

On a fresh system (or after removing `~/.local/bin/bts`), run the pipe-mode install. All 12 tools install via mise, but `bts` itself is never copied to a `PATH`-accessible location. `command -v bts` returns nothing.

### Isolate

`install.sh` has four steps:
1. Ensure mise is installed
2. Locate/copy `mise.toml` (with pipe-mode fallback)
3. `mise install` — installs all 12+ tools from the manifest
4. Print done message

Step 4 references `bts init` in its message, but `bts` is not available because it was never installed. The script never touches `bin/bts`.

### Hypothesize

**Hypothesis**: Adding a step 4a that copies `bin/bts` to `~/.local/bin` (with pipe-mode fallback, matching the existing pattern for `mise.toml`) will make `bts` available on `PATH` immediately.

**Falsification test**: After adding the copy step, `command -v bts` returns a path and `bts --version` prints the version.

### Verify

**Confirmed**: `install.sh` is the single point of entry. It copies `mise.toml` but not `bin/bts`. The pipe-mode fallback pattern (fetch from `REPO_RAW` when not co-located) is already established and reusable.

**Fix approach**: Add a step after `mise install` that:
1. Creates `~/.local/bin/` if it doesn't exist
2. Copies `bin/bts` if co-located, or fetches from `REPO_RAW/bin/bts` in pipe mode
3. Makes it executable
4. Reports the installed path

Additionally:
- Add `bts` to the `[tasks.doctor]` check list so users can verify their install
- Add a structural test in `mise run test` to verify `install.sh` installs `bts`

### Risk level: Low

Copies a single shell script to a standard XDG user bin directory. No dependencies, no compilation, no system-level changes. `~/.local/bin` is already on `PATH` on most modern Unix systems.

## Resolution

**Fixed:** 2026-06-22
**Root cause confirmed:** `install.sh` only installed the mise toolchain; it never copied `bin/bts` to a PATH-accessible location. Users had all dependencies but no `bts` command.
**Fix applied:**
1. Added step 5 to `install.sh`: copies `bin/bts` to `~/.local/bin/` with pipe-mode fetch fallback (reuses existing `REPO_RAW` + `HERE` pattern)
2. Added `bts` to `[tasks.doctor]` check list in `mise.toml`
**Hardening added:** 3 structural assertions in `mise run test`:
1. `install.sh` has `~/.local/bin/bts` copy step
2. `install.sh` has pipe-mode fetch fallback for `bts`
3. Doctor check includes `bts`
**Evidence:** All 15 tests pass (`mise run test` — 15 passed, 0 failed). `bts --version` prints `0.2.1`. `mise run doctor` reports `ok bts`.
**Commit:** `feat(installer): install bts itself to ~/.local/bin during install.sh`
