# BUG-20260618T044939: `bts compress --stdin` rejects the flag it advertises

## Problem

`bts compress --stdin` fails with "Unknown flag: --stdin" despite the usage message
listing `[--stdin]` as a valid flag.

- **Actual behavior**: Fails at flag parsing, exits 1, never reaches the compression
  engine. Both piped input and empty stdin produce the same error.
- **Expected behavior**: `--stdin` should be accepted as a no-op flag (stdin is
  already detected via `[ ! -t 0 ]`) or as a force flag to override TTY detection.
- **Reproduction**:
  ```
  echo "text" | bts compress --stdin     # fails
  bts compress --stdin                    # fails (no pipe, TTY)
  echo "text" | bts compress             # works (pipe detection)
  ```

- **Affected integration**: The pi `bts_compress` custom tool always passes `--stdin`
  to the bts dispatcher, and always hits this failure. The extension degrades
  gracefully (returns text uncompressed), so the user-visible impact is that
  compression is silently skipped.

## Root Cause Analysis

The compress verb was added in commit 8142ac6 (`feat(bts): config profiles, compress,
and regret-learning map`). The flag-parsing case statement in `bts_compress` handles
`--budget` and `--budget=`, then falls through to a wildcard `-*)` that rejects
any unknown flag. The `--stdin` flag appears in the usage message at the error output
site and in the help text, but was never added as a case arm.

The documentation/implementation mismatch has existed since the feature was introduced
— this is not a regression.

**Contributing factors**:
- The epic's verification criteria (e07) only tested piped stdin (`echo input | bts
  compress`), not the explicit `--stdin` flag
- The pi extension was written assuming `--stdin` worked, masking the bug with a
  catch-and-fallback pattern

**Risk**: Low — compression degrades gracefully (returns uncompressed text) in the pi
extension. No data loss or incorrect output. Only broken for CLI users who explicitly
pass `--stdin`.

1. **RED**: `echo "test" | bts compress --stdin` exits 0 (currently exits 1).
   **GREEN**: Add `--stdin)` arm to the flag-parsing case statement that advances
   `shift` without setting any variable (no-op — pipe detection already handles it).
   **verify**: `echo "test" | ./bin/bts compress --stdin 2>&1; [ $? -eq 0 ]`

2. **RED**: `bts compress --stdin --budget 512` parses both flags (currently fails on
   `--stdin` before ever seeing `--budget`).
   **GREEN**: Same change as above — the case arm must `shift` and `continue` so
   subsequent flags are parsed.
   **verify**: `echo "test" | ./bin/bts compress --stdin --budget 512 2>&1; [ $? -eq 0 ]`

3. **RED**: `bts compress --budget 512 --stdin` works regardless of flag order.
   **GREEN**: Same `--stdin` case arm — order-independence is inherent if both flags
   are handled.
   **verify**: `echo "test" | ./bin/bts compress --budget 512 --stdin 2>&1; [ $? -eq 0 ]`

**REFACTOR**: None — one-line addition to the case statement.

## Resolution

Fixed 2026-06-18: Added `--stdin` case arm to the `bts_compress` flag-parsing case
statement in `bin/bts`, between `--budget=*)` and the `-*)` wildcard.
