# BUG-20260622T193000: fd install fails on Intel Mac (darwin/amd64)

## Problem

Running `install.sh` on an Intel Mac fails with:

```
mise ERROR Failed to install aqua:sharkdp/fd@latest: unsupported env: darwin/amd64
(supported: ["linux", "darwin/arm64", "windows"])
```

**Actual**: `fd` is left as `missing` in `mise list`; all other tools install fine.

**Expected**: `fd` installs successfully on all platforms, including Intel Mac.

**Reproduce**: On any darwin/amd64 machine, run `bash install.sh` (or `mise install` with this project's `mise.toml`).

## Root Cause Analysis

### Reproduce

On any darwin/amd64 machine, run `mise install` with the project's `mise.toml`. The bare `fd = "latest"` entry resolves to `aqua:sharkdp/fd`, which errors: `unsupported env: darwin/amd64 (supported: ["linux", "darwin/arm64", "windows"])`. All other tools install successfully; only `fd` is left as `missing`.

### Isolate

### The chain

1. `mise.toml` declares `fd = "latest"` — a bare name. mise resolves it to `aqua:sharkdp/fd` (the first backend in `mise registry fd`).
2. The aqua backend for `sharkdp/fd` explicitly rejects darwin/amd64 — it only supports linux, darwin/arm64, and windows.
3. Aqua errors out hard; mise does NOT fall through to the next backend (`asdf`, then `cargo`). The install aborts.

### Hypothesize

**Hypothesis**: The aqua backend errors because sharkdp/fd no longer publishes Intel Mac binaries. Aqua's platform support list is correct — sharkdp/fd v10.4.0+ only ships `aarch64-apple-darwin` (ARM Mac).

**Falsification test**: If fd v10.3.0 has Intel Mac binaries, and v10.4.0+ does not, and the changelog confirms the intentional drop, then the hypothesis is confirmed.

### Why aqua lacks darwin/amd64

This is NOT a mise or aqua bug — it's an **intentional upstream decision by sharkdp/fd**. fd v10.4.0 (2026-03-07) bumped its minimum Rust version to 1.90.0 and the changelog explicitly states: *"Notably, this means dropping fully support for intel Mac."* GitHub releases confirm: v10.3.0 (2025-08-26) was the LAST release with `x86_64-apple-darwin` binaries; v10.4.0+ ship only `aarch64-apple-darwin`.

The aqua registry correctly reflects this by marking darwin/amd64 as unsupported.

### Verify

**Confirmed root cause**: sharkdp/fd intentionally dropped Intel Mac support in v10.4.0 (2026-03-07). Evidence:
- GitHub releases: v10.3.0 (2025-08-26) ships `fd-v10.3.0-x86_64-apple-darwin.tar.gz`; v10.4.0+ has no Intel Mac binary across all 3 releases (v10.4.0, v10.4.1, v10.4.2)
- Changelog: "Minimum required rust version has been increased to 1.90.0. Notably, this means dropping fully support for intel Mac."
- Aqua registry: correctly reflects this by marking darwin/amd64 as unsupported

**Fix approach**: Pin fd to `"github:sharkdp/fd" = "10.3.0"` — the last release with universal macOS support. This gets a prebuilt binary on all platforms without requiring a Rust toolchain.

### Secondary finding: GitHub backend at `latest` also fails

The issue reporter's suggested fix (`"github:sharkdp/fd" = "latest"`) would also fail on Intel Mac because the latest GitHub release (v10.4.2) also lacks Intel Mac binaries. The GitHub backend would encounter the same missing-platform-asset problem.

### Contributing factors

- mise does not fall through to the next backend when the first one errors (aqua → asdf → cargo chain breaks at aqua)
- `bat` (also from sharkdp, also bare `bat = "latest"`) does NOT have this issue — `bat` still ships `x86_64-apple-darwin` binaries in its latest release
- No other bare-name tools in `mise.toml` are affected; `ripgrep`, `bat`, `eza`, `fzf`, `jq`, `gh`, `direnv`, `shellcheck`, `shfmt`, `bats` all resolve to backends with darwin/amd64 support

### Risk level: Low

One manifest line change. v10.3.0 is a well-tested, full-featured release of `fd`. The differences between v10.3.0 and v10.4.x are minor (a `--ignore-contain` flag, Windows/MSVC linking changes, and ignore-library bugfixes — none affect core search behavior).

## Resolution

**Fixed:** 2026-06-22
**Root cause confirmed:** sharkdp/fd intentionally dropped Intel Mac support in v10.4.0 (Rust 1.90.0 requirement). Bare `fd = "latest"` resolved to aqua backend which rejects darwin/amd64.
**Fix applied:** Changed `fd = "latest"` to `"github:sharkdp/fd" = "10.3.0"` — last release with x86_64-apple-darwin binaries.
**Hardening added:** Two structural assertions in `mise run test`:
1. fd must use explicit `github:sharkdp/fd` backend (not bare name → aqua fallback)
2. fd must be pinned to `10.3.0` (not `latest`, which lacks Intel Mac binaries)
**Evidence:** All 12 tests pass (`mise run test` — 12 passed, 0 failed). `mise install fd` succeeds. `fd --version` reports working binary.
**Commit:** `fix(toolchain): pin fd to github:sharkdp/fd@10.3.0 for Intel Mac support`
