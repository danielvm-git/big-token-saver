# BUG-20260622T180000: Release workflow always red — npm publish runs on every push without auth

## Problem

Every push to `main` triggers the Release workflow, which always fails at the "Publish to npm" step. The failure makes the entire workflow appear broken (red ❌) even though the actual release logic (semantic-release + binary builds) works correctly.

### Actual behavior

- Release workflow runs on every push to main
- `npm publish` step always attempts to publish, reading the latest git tag
- Fails with `ENEEDAUTH` because no `NPM_TOKEN` secret is configured
- Even if NPM_TOKEN were set, it would fail with "version already exists" on non-release pushes (re-publishing the same version)

### Expected behavior

- Release workflow is green on pushes that don't trigger a new release (no `feat:` or `fix:` commits)
- npm publish only runs when a new version was actually created by semantic-release
- Missing NPM_TOKEN produces a clear skip message, not a red failure

### Reproduction

Any push to main. Example: `gh run view 27969662581` — CI passes, Release fails.

## Root Cause Analysis

The Release workflow treats npm publishing as an unconditional post-release step rather than a conditional action gated on whether semantic-release actually created a new version.

**The code path:**
1. Build job compiles bts-map for 3 platforms (always succeeds)
2. Release job runs `npx semantic-release` — analyzes commits; creates a new tag ONLY when there are `feat:` or `fix:` commits since the last tag
3. "Publish to npm" step runs `git describe --tags --abbrev=0` — reads the latest tag (which may be from a previous release, not necessarily one just created)
4. If a tag exists (which it always will after the first release), attempts `npm publish` — fails because `NPM_TOKEN` secret is not set

**Why the current design fails:**

- **No coupling between semantic-release output and npm publish.** The npm publish step doesn't know whether semantic-release just created a new version or did nothing.
- **No auth guard.** The step doesn't check whether NPM_TOKEN is available before attempting publish.
- **Publish on every push.** Even with auth, re-publishing the same version would fail.

**Risk level:** Low — fix is contained to the Release workflow YAML; no code changes needed.

## TDD Fix Plan

### Cycle 1: npm publish is skipped when no new release was created

1. **RED**: Verify that a push with only non-releasing commits (e.g., `test:`, `chore:`) results in a green Release workflow (npm publish step skipped).

   **GREEN**: Capture the latest git tag before running semantic-release. After semantic-release, compare with the new latest tag. Only proceed to npm publish if the tags differ (a new release was created). If tags are the same, print a skip message and exit 0.

   **verify**: Push a `test:` commit → Release workflow exits green, npm publish step shows "No new release — skipping npm publish"

### Cycle 2: npm publish is skipped when NPM_TOKEN is not configured

2. **RED**: With no NPM_TOKEN secret set and a valid release, the workflow is still green (publish step skipped with a clear message).

   **GREEN**: Add a guard at the start of the Publish to npm step: if `NPM_TOKEN` is empty or unset, print "NPM_TOKEN not configured — skipping npm publish" and exit 0. This guard runs BEFORE the tag comparison so it covers both cases.

   **verify**: Push a `fix:` commit (triggers release) with no NPM_TOKEN → Release workflow shows "NPM_TOKEN not configured — skipping npm publish", exits green

**REFACTOR**: None needed — the changes are isolated to the Publish to npm step's shell script.
