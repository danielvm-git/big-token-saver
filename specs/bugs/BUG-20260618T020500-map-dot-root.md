# BUG-20260618T020500: `bts map` returns "no source files" when run from project root

## Problem

`bts map` (and the underlying `bts-map` binary) returns "no source files found under ."
when invoked without a path argument from the project root, despite the project
containing Rust, TypeScript, Python, Go, and Java source files.

- **Actual behavior**: `bts map` prints "bts-map: no source files found under ." and exits 0
- **Expected behavior**: A ranked file map listing source files (crates/bts-map/src/*.rs,
  tests/fixtures/*.ts, etc.)
- **Reproduction**:
  ```
  cd /path/to/big-token-saver
  bts map            # no output
  bts-map .          # "no source files found under ."
  bts-map "$(pwd)"   # works — produces full map
  bts-map ./crates   # works — produces map for crates/ subtree
  ```

- **Affected integration**: The pi `bts_map` tool runs `bts map` without arguments,
  always hitting this silent failure. The agent degrades by reading files raw
  instead of getting a ranked map, losing the structural overview.

## Root Cause Analysis

The `is_excluded()` filter in the directory-walk loop uses a hidden-directory
heuristic (`name.starts_with('.')`) to skip directories like `.git`, `.direnv`, etc.
This check has no depth guard — it applies to the walk root as well as all
descendants. When the user passes `.` as the scan root, `walkdir` returns the
root entry with `file_name() == "."` and `depth() == 0`. The hidden-directory
check matches `"."` (it starts with `.`), and `filter_entry` excludes the root
*and all its children*, producing an empty tag set — hence "no source files found."

Other paths work because their root component doesn't start with `.`:
- Absolute paths (e.g., `/home/user/project`) → root `file_name()` is `"project"`
- Relative subdirectories (e.g., `./src`) → root `file_name()` is `"src"`

The bug has existed since the directory-walk was introduced (commit 8142ac6).
It is not a regression — the default-path case was simply never exercised in
the test suite (all tests use absolute fixture paths or in-memory strings).

**Contributing factors**:
- The existing test suite validates tag extraction, ranking, feedback, budget,
  and snapshot output, but has no integration test for the directory-walk loop
  itself (no test that calls `collect_tags` on a real directory)
- `filter_entry` is a blunt instrument — if the root is filtered, the entire
  subtree is silently dropped

**Risk**: Medium — the map is the primary structural overview tool for the pi agent.
Without it, the agent falls back to raw file reads, losing the PageRank ordering
and regret-learning feedback that make `bts map` valuable.

## Resolution

**Fixed:** 2026-06-18
**Root cause confirmed:** `is_excluded()` had no depth guard on its `starts_with('.')` check, so `walkdir`'s root entry (`file_name() == "."`) was excluded along with all children when the user passed `.` as the scan root.
**Fix applied:** Added `entry.depth() > 0 &&` guard to the hidden-directory check in `is_excluded()`. The walk root (depth 0) is never excluded by the dot-prefix heuristic, while hidden child directories at depth ≥ 1 remain excluded.
**Hardening added:** 3 new integration tests: `walk_from_dot_produces_tags`, `walk_from_hidden_root_produces_tags`, `hidden_children_are_excluded` — exercising the directory-walk loop with real temp directories.
**Evidence:** 19/19 tests pass (16 original + 3 new), `bts-map .` and `bts map` both produce non-empty output from project root.
